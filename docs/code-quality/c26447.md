# <a name="c26447-dont_throw_in_noexcept"></a>C26447 DONT_THROW_IN_NOEXCEPT

Funkcja jest zadeklarowana, **`noexcept`** ale wywołuje funkcję, która może zgłaszać wyjątki.

Podstawowe wytyczne dotyczące języka C++: [F. 6: Jeśli funkcja nie może zgłosić, zadeklaruj ją noexcept](https://github.com/isocpp/CppCoreGuidelines/blob/master/CppCoreGuidelines.md#f6-if-your-function-may-not-throw-declare-it-noexcept).

Ta reguła zmienia inną regułę, [C26440 DECLARE_NOEXCEPT](c26440.md), która próbuje znaleźć funkcje, które są dobrymi kandydatami do oznaczenia jako **`noexcept`** . W tym przypadku pomysłem jest to, że po oznaczeniu jakiejś funkcji jako **`noexcept`** , musi ona zachować swoją umowę, nie wywołując innego kodu, który może zgłaszać wyjątki.

- Kompilator języka Microsoft C++ już obsługuje proste naruszenia, takie jak **`throw`** instrukcje w treści funkcji (zobacz [C4297](/cpp/error-messages/compiler-warnings/compiler-warning-level-1-c4297)).
- Reguła koncentruje się tylko na wywołaniach funkcji. Flaguje elementy docelowe, które nie są **`constexpr`** i które mogą potencjalnie zgłosić wyjątki; innymi słowy, nie są oznaczone jawnie jako niepodlegające przerzucaniu przy użyciu **`noexcept`** , **`__declspec(nothrow)`** , **throw ()**.
- Funkcje docelowe generowane przez kompilator są pomijane w celu zmniejszenia szumu, ponieważ specyfikacje wyjątków nie zawsze są dostarczane przez kompilator.
- Moduł sprawdzania pomija również specjalne rodzaje funkcji docelowych, które powinny być zaimplementowane jako **`noexcept`** ; Ta reguła jest wymuszana przez [C26439 SPECIAL_NOEXCEPT](c26439.md).
